# Writer Agent ビジュアル連携ガイド

> Writer AgentがVisualizer Agentと連携してビジュアル付き記事を制作するための手順書
> 最終更新: 2026-03-05（v2: VISUALタグ新フォーマット・3D指示・テーマ別テンプレート追加）

## 1. なぜビジュアルが必要か

### エンゲージメントへの影響

| 指標 | テキストのみ | ビジュアル付き | 改善率 |
|------|------------|--------------|--------|
| 記事CTR（検索結果） | ベースライン | +50-100% | 2倍 |
| SNSエンゲージメント | ベースライン | +150-300% | 2.5-4倍 |
| 記事滞在時間 | ベースライン | +30-50% | 1.3-1.5倍 |
| 被リンク獲得率 | ベースライン | +2-3倍 | 情報源として引用されやすい |
| AI引用率(GEO) | ベースライン | +20-30% | 図解がパッセージに含まれると引用されやすい |

### SEOへの効果

- **画像検索からの流入**: Google画像検索結果に表示される
- **リッチスニペット**: 画像付き検索結果はCTRが高い
- **ページ滞在時間**: ビジュアルが多い記事はスクロール深度が深い
- **alt属性**: キーワードを含むalt属性はSEOに直接貢献
- **被リンク**: 図解やインフォグラフィックは他メディアに引用されやすい

## 2. ビジュアル指示タグの仕様

### 新フォーマット: VISUALタグ（推奨）

```html
<!-- VISUAL: [type] | テーマ: [テーマの説明] | スタイル: [スタイルキーワード] | 比率: [アスペクト比] -->
```

### 旧フォーマット: IMAGEプレースホルダー（互換対応）

```html
<!-- IMAGE: [タイプ]
  説明: [何を視覚化するか]
  ...
-->
```

両方のフォーマットに対応するが、新規記事は**VISUALタグ**を使用する。

---

### パラメータ一覧

#### type（必須）

| type | 用途 | 生成ツール（実機能） |
|------|------|-----------|
| `hero` | 記事冒頭のヒーロー画像 | **Nano Banana Pro**（Google AI Studio API） |
| `diagram` | フローチャート・構成図 | Mermaid / HTML/CSS |
| `chart` | グラフ・データビジュアライゼーション | HTML/CSS |
| `infographic` | インフォグラフィック | HTML/CSS |
| `3d` | 3Dレンダリング風ビジュアル | **Nano Banana Pro** |
| `social` | SNSカード画像 | **Nano Banana Pro** |
| `comparison` | 比較表・マトリクス（視覚的） | HTML/CSS |
| `timeline` | タイムライン | Mermaid / HTML/CSS |
| `quote` | 引用カード・ハイライト | HTML/CSS |
| `cta` | CTA前ビジュアル | **Nano Banana Pro** |

#### テーマ（必須）

ビジュアルで表現したい内容の具体的な説明。抽象的すぎる指示は避ける。

悪い例:
```html
<!-- VISUAL: hero | テーマ: AIについて | スタイル: ダーク | 比率: 16:9 -->
```

良い例:
```html
<!-- VISUAL: hero | テーマ: 19個のAIエージェントが協調して朝会を行うシステムの全体像 | スタイル: ダーク3D アイソメトリック | 比率: 1.91:1 -->
```

#### スタイル（推奨）

| キーワード | 意味 |
|-----------|------|
| `ダーク3D` | 黒背景+3Dレンダリング風 |
| `アイソメトリック` | 斜め上からの3D構成 |
| `ダッシュボード風` | UI/ダッシュボード表現 |
| `ガラスモーフィズム` | 透明感のあるフローティングカード |
| `ホログラム` | データが光る表現 |
| `ミニマル` | 極限にシンプル、余白多め |
| `Nothing Tech風` | Nothing Phone的なドットマトリクス+グリフ |
| `Apple風` | Apple的な余白とクリーンさ |
| `回路基板` | サーキットボードパターン |
| `ネットワーク` | ノード接続のビジュアル |
| `キーノート風` | プレゼンスライドのような大文字+余白 |

#### 比率（推奨）

| 比率 | 用途 |
|------|------|
| `1.91:1` | ブログヒーロー、OGP、noteサムネイル |
| `16:9` | インライン画像、Xポスト |
| `1:1` | Instagram正方形フィード |
| `4:5` | Instagramカルーセル |
| `9:16` | Instagramストーリーズ/リール |
| `2:3` | Pinterestピン |

#### 展開（任意）

マルチプラットフォーム展開時に指定:
```html
<!-- VISUAL: hero | テーマ: [テーマ] | スタイル: ダーク3D | 比率: 1.91:1 | 展開: blog, note, x, instagram, linkedin -->
```

## 4. 記事内のビジュアル配置ルール

### 最低枚数

| 記事タイプ | 最低枚数 | 理想枚数 |
|-----------|---------|---------|
| ブログ記事（2,000語以上） | 3枚 | 5枚 |
| ブログ記事（1,000語未満） | 2枚 | 3枚 |
| note記事 | 2枚 | 4枚 |
| ハウツーガイド | 4枚 | 6枚以上 |
| ニュース・速報 | 1枚 | 2枚 |

### 配置位置の原則

```
記事構造:

[H1 タイトル]
<!-- VISUAL: hero -->                    <-- 必須: ヒーロー画像

[導入パラグラフ 200-300語]

[H2 セクション1]
<!-- VISUAL: diagram or chart -->        <-- 推奨: セクション図解
[本文 300-500語]

[H2 セクション2]
<!-- VISUAL: 3d or infographic -->       <-- 推奨: 視覚的ブレイク
[本文 300-500語]

[H2 セクション3]
[本文 300-500語]
<!-- VISUAL: chart or comparison -->     <-- 推奨: データビジュアル

[H2 まとめ / CTA]
<!-- VISUAL: cta -->                     <-- 推奨: CTA前ビジュアル
[CTA テキスト]
```

### 配置の禁止事項

1. **画像の連続配置**: 2つのVISUALタグ間に最低200語のテキストを挟む
2. **H3内の大画像**: H3セクション内にフルサイズ画像は入れない（図解は可）
3. **記事末尾の画像放置**: 最後のVISUALタグの後にテキスト（CTA等）がない状態
4. **ヒーロー画像の省略**: いかなる記事でもヒーロー画像は必ず入れる

**黄金律: 500字以上テキストが続いたら、ビジュアルの挿入を検討する**

## 5. 指示の書き方

### Do（推奨）

- テーマを具体的に書く: 「中小企業のDX推進フローを5ステップで表現」
- データがあるなら数字を入れる: 「ROAS 350%の改善を示すBefore/Afterグラフ」
- 感情・印象を添える: 「テック企業の未来感。冷たいが知的なイメージ」
- 記事内の文脈と接続する: 「直前のパラグラフで説明した3層アーキテクチャの図解」

### Don't（禁止）

- 曖昧すぎる指示: 「AIの画像」「いい感じの図」
- ブランド外の指示: 「明るくてポップな画像」「パステルカラーで」
- ストック写真の依頼: 「ビジネスマンが握手している画像」
- テキスト直接生成の依頼: 「画像内に日本語で'AIの未来'と入れて」（崩れるため禁止）

## 6. テキストオーバーレイが必要な場合

画像内にテキストが必要な場合は、Visualizer Agentが後処理で追加する。タグ内で明示する:

```html
<!-- VISUAL: sns | テーマ: "2026年、AIエージェントは会社になる"のテキストカード | スタイル: タイポグラフィ主導 | 比率: 16:9 | テキスト: "2026年、AIエージェントは会社になる" | フォント: Inter 700 -->
```

日本語テキストはAI生成不可（崩れるため）。英語テキストはIdeogram 3.0で98%精度。

## 8. alt属性の書き方

### ルール

1. **具体的に書く**: 「画像」「図」などの曖昧な表現は避ける
2. **コンテキストを含める**: 記事内での役割がわかるように書く
3. **キーワードを含める**: SEO対象キーワードを自然に含める
4. **100文字以内**: 長すぎるalt属性は逆効果
5. **装飾的画像は空文字**: 意味を持たない装飾画像は `alt=""` にする

### 良い例

```markdown
![AIエージェントシステムの全体構成図。CEO Agentを中心に18の専門エージェントが放射状に接続される構造を示す3Dイラスト](/images/blog/2026-03/diagram-agent-architecture-01.webp)

![AI画像生成ツール5社のコスト比較。SDXLが$0.006で最安、Ideogramが$0.06で最高コスト](/images/blog/2026-03/infographic-tool-cost-comparison.webp)
```

### 悪い例（使わないこと）

```markdown
![画像1](/images/blog/image1.webp)
![図](/images/blog/chart.webp)
```

## 9. frontmatterの画像関連フィールド

```yaml
---
title: "記事タイトル"
date: "2026-03-05"
description: "メタディスクリプション（120文字以内）"
author: "writer"
tags: ["AI", "エージェント", "コンテンツマーケティング"]
type: "article"
heroImage: "/images/blog/2026-03/hero-[slug].webp"
ogImage: "/images/blog/2026-03/og-[slug].png"
---
```

- `heroImage`: WebP形式。ブログ内表示用。
- `ogImage`: PNG形式。SNSシェア時に使われる（WebPだとSNSクローラーが認識しない場合あり）。
- **両方設定する**。片方だけでは不十分。

## 10. Visualizer Agentとの連携フロー

```
Writer Agent                         Visualizer Agent
     |                                      |
     |  1. 記事アウトライン作成               |
     |     (VISUALタグを含む)                |
     |                                      |
     |  2. 記事本文を執筆                    |
     |     (VISUALタグ確定)                  |
     |                                      |
     | -----> 記事をVisualizer Agentに送る ---->  |
     |                                      |
     |                          3. VISUALタグを解析
     |                          4. プロンプト生成・画像制作
     |                          5. 3D/図解/AI画像を生成
     |                          6. 画像最適化
     |                          7. VISUAL-SPEC.md作成
     |                                      |
     | <---- ビジュアル一式 + 仕様書を受け取る <----  |
     |                                      |
     |  8. 画像をMarkdownに埋め込み           |
     |  9. frontmatterのheroImage/ogImage設定 |
     | 10. alt属性を最終調整                  |
     | 11. 完成版の記事をpush                 |
```

### ポイント

- Writer Agentはビジュアルの「何が必要か」を指示する（内容・目的）
- Visualizer Agentが「どう見せるか」を決定する（スタイル・ツール選択・レイアウト）
- 最終的な記事への埋め込みはWriter Agentが責任を持つ
- alt属性テキストはWriter Agentが記事のコンテキストに合わせて最終調整する

## 11. プラットフォーム別ビジュアル展開

### 1記事から生成するビジュアル一覧

```
1本の記事 --> 以下のビジュアルを生成

ブログ用:
  +-- hero-[slug].webp        (1200x630)
  +-- og-[slug].png           (1200x630)

  +-- diagram-[slug]-02.webp  (1200x675)

  +-- x-card-[slug].png       (1200x675)
  +-- note-thumb-[slug].png   (1280x670)

  +-- linkedin-[slug].png     (1200x627)
```

### 各プラットフォームの注意点

| プラットフォーム | 注意点 |
|---------------|--------|

| Pinterest | 縦長必須（2:3以上）。インフォグラフィックが効果的 |

## 12. よくある質問

### Q: VISUALタグを入れ忘れた場合は？
A: Visualizer Agentが記事をレビューし、不足箇所に追加提案する。ただしWriter Agent側で意図的に入れるのが理想。

### Q: 既存画像を使いたい場合は？
A: タグ内に `既存: [パス]` を追加:
```html
<!-- VISUAL: screenshot | テーマ: ダッシュボードのスクリーンショット | 既存: ./images/dashboard.png | 加工: ダークフレーム追加 -->
```

### Q: 日本語テキストは画像に入れられるか？
A: AI画像生成ツールでの日本語テキスト直接生成は禁止。後処理でオーバーレイを追加する。英語テキストはIdeogram 3.0で98%精度。

### Q: SNS向け画像だけ必要な場合は？
A: 記事末尾にSNS展開用セクションを設ける:
```html
<!-- SNS展開 -->
<!-- VISUAL: sns | テーマ: [記事タイトル要約] | スタイル: タイポグラフィ | 比率: 16:9 | プラットフォーム: x -->
<!-- VISUAL: sns | テーマ: [キーインサイト] | スタイル: ミニマル | 比率: 1:1 | プラットフォーム: instagram -->
```

## 13. よくあるミスと回避方法

### ミス1: VISUALタグなしで記事を書き切る
**回避**: アウトライン段階でVISUALタグを先に配置する。後から入れると位置が不自然になる。

### ミス2: alt属性を後回しにして忘れる
**回避**: 画像を埋め込む時点で必ずalt属性を書く。空のalt属性は公開しない。

### ミス3: OGP画像をWebPで設定してしまう
**回避**: OGP画像は必ずPNG。frontmatterの`ogImage`は常に`.png`ファイルを指定。

### ミス4: 画像サイズが大きすぎてページ速度を低下させる
**回避**: ヒーロー画像200KB以下、インライン150KB以下を厳守。

### ミス5: 全ての記事に同じスタイルの画像を使い回す
**回避**: ヒーローのプロンプトパターン（A-1, A-2, A-3）を記事テーマに応じて使い分ける。
