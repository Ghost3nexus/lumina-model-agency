# 開発品質ゲート — Dev Agent 必須プロセス

> 2026-03-11 CEO承認。コンテンツ品質ゲート（blog-publishing-pipeline.md）と同等の厳格さを開発にも適用する。
> 「動くものを最速で」は「壊れたものを最速で出す」ではない。

---

## 根本原則

1. **読んでから書く** — コードを1行も変更する前に、対象ファイルとその依存関係を全て読む
2. **変更したら検証する** — 「ビルドが通る」は検証ではない。ユーザーフローで動作確認する
3. **コミットするまで未完了** — 未コミットの変更は存在しないのと同じ。作業セッション終了時に必ずコミット or リバート
4. **二重生成パイプラインは疑う** — AI出力を別のAIに食わせる構成は品質劣化のリスク。単一パスで済むなら単一パスにする

---

## Phase 1: 着手前（必須）

### コードリーディング

変更対象のファイルに触れる前に、以下を実行:

```
1. 対象ファイルを全行読む（Read tool）
2. そのファイルがimportしているモジュールを特定
3. そのファイルをimportしている側（呼び出し元）を特定（Grep tool）
4. 影響範囲を分類:
   - isolated: 1ファイル完結、他から参照されていない
   - moderate: 2-5ファイルが依存
   - wide: 共有ユーティリティ/APIクライアント/サービス層
5. wide の場合 → 変更後に全依存元の動作確認が必須
```

### 既存動作の理解

```
変更前に以下を言語化する（ログ or コメントで記録）:
- この関数/モジュールは今何をしているか（1-2文）
- データの入力と出力は何か
- どこから呼ばれ、結果はどう使われるか
- 現在の既知の問題は何か
```

---

## Phase 2: 実装中（必須）

### 変更ルール

- **1つの変更 = 1つの目的**: 品質改善とUI変更を同じコミットに混ぜない
- **パイプライン変更は特に慎重に**: サービスA → サービスBのデータフローを変える場合、A単体とB単体の両方で動作確認
- **AIサービス連携の鉄則**:
  - AI出力を別のAIの入力にする場合、中間出力を必ず目視確認
  - 色味・解像度・フォーマットの劣化がないか検証
  - 単一パスで同じ結果が出せないか常に検討する

### コミット頻度

```
- 機能単位でこまめにコミット（1セッションで1コミット以上）
- 「あとでまとめてコミット」は禁止
- 作業中断時も必ずWIPコミット or stash
```

---

## Phase 3: 検証（必須 — スキップ不可）

### ビルド検証

```bash
# 必ず実行
npm run build  # or tsc && vite build
# エラー0件を確認
```

### 機能検証（ビルド通過だけでは不十分）

```
1. 変更した機能のユーザーフローを端から端まで確認
   - 入力 → 処理 → 出力 の全ステップ
   - エッジケース（空入力、大きなファイル、エラー時）

2. 隣接機能の回帰チェック
   - 変更したモジュールを使っている他の機能が壊れていないか
   - wide 影響範囲の場合、全依存元を確認

3. 画像生成パイプラインの場合（Lumina等）:
   - 出力画像を目視確認（白飛び、色ずれ、解像度劣化）
   - 変更前の出力と比較（A/Bテスト的に）
   - 複数パターンで試す（異なるガーメント、異なるモデル）
```

### 本番デプロイ前チェックリスト

```
□ ビルドがエラー0件で通る
□ 変更した機能が正しく動く（目視確認済み）
□ 隣接機能が壊れていない（回帰テスト済み）
□ 全変更がコミット済み（git status クリーン）
□ セキュリティ変更がある場合 → Security Agent レビュー済み
□ Vercel Preview URL で動作確認済み（Vercelの場合）
□ wide 影響範囲の変更 → dev:reviewer サブエージェントでレビュー済み
```

---

## Phase 4: デプロイ（ゲート通過後のみ）

```
1. git push origin main（auto-deploy）
2. デプロイ完了を確認（Vercel/Railway のステータス）
3. 本番URLで最終動作確認
4. 問題があれば即 revert（git revert、force pushではない）
```

---

## Claude Code Review 連携（2026-03-11〜）

GitHub PRベースの開発フローでは、Claude Code Review を活用する:

- PR作成時に自動でマルチエージェントレビューが走る
- バグ検出、セキュリティ脆弱性、ロジックエラー、エッジケースを並列チェック
- 信頼度80以上の指摘のみコメント（ノイズ低減）
- **Code Reviewの指摘は無視しない** — 全件対応 or 対応不要の理由を明記

---

## dev:reviewer サブエージェント必須化

以下の条件で `dev:reviewer` の生成が**必須**:

| 条件 | 理由 |
|------|------|
| 本番デプロイを含むPR | 第三者の目でコードと動作を確認 |
| wide 影響範囲の変更 | 見落としリスクが高い |
| AIサービス連携の変更 | パイプライン相互作用の検証 |
| 3ファイル以上の同時変更 | 複合的な影響の確認 |

dev:reviewer の責務:
1. diff を読んで変更意図を理解
2. ビルド通過を確認
3. 影響を受けるユーザーフローを特定
4. 最低1つのユーザーフローを端から端まで検証
5. 問題があれば親Devに報告（デプロイブロック）

---

## 違反時の対応

| 違反 | 結果 |
|------|------|
| 未コミットのまま「完了」報告 | タスクをrejected。即座にコミット or リバート |
| 検証なしでデプロイ | インシデントとして記録。原因分析をJournalに残す |
| コードリーディングなしで変更 | 変更をリバート。読み直してからやり直し |
| 画像パイプライン変更で目視確認なし | デプロイブロック。出力画像の比較を提出 |

## アンチパターン集（過去の失敗から）

| アンチパターン | 何が起きたか | 正しい対処 |
|--------------|------------|----------|
| **コード変更してコミットせず** | 6ファイル修正が本番未反映。白飛び「治らない」と報告 | 変更→即コミット→push |
| **AI出力を別AIに食わせる** | VTON出力→Gemini再生成で白飛び。ピクセル精度が台無し | 単一パスで完結させる。二重生成は品質劣化 |
| **既存コードを読まずに機能追加** | VTON+Geminiの相互作用を理解せずにVTON統合 | 依存関係を全て読んでから着手 |
| **間違ったデプロイ先にpush** | lumina-appにデプロイ。本番はllumina-fashion-studio | デプロイ先を事前確認。.vercel/project.json を読む |
| **ビルド通過=完了と判断** | ビルドは通るがユーザーフローが壊れている | ユーザーフローの端から端まで確認 |
