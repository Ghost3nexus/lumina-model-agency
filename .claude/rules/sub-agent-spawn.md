# サブエージェント自律生成ルール

> 2026-03-09 CEO承認。エージェントは自身がマネージャーとなり、
> 負荷分散のためにサブエージェントを自発的に生成してよい。

---

## 原則

1. **全エージェントがマネージャーになれる** — タスク負荷が高い場合、自身の判断でサブエージェントを生成できる
2. **親エージェントが全責任を負う** — サブエージェントの成果物・品質・セキュリティは親が最終レビューする
3. **承認不要で即時生成** — CEO/Councilへの事前承認は不要。事後にJournalで記録すればよい
4. **一時的な存在** — サブエージェントはタスク完了後に消滅する。恒久化が必要なら Council で正式エージェント昇格を議論する

---

## 生成条件（いつサブエージェントを作るか）

| 条件 | 例 |
|------|---|
| **1日3タスク上限を超える** | Dev が5件抱えている → 2件をサブに委譲 |
| **並行実行が必要** | セキュリティ監査とAPI実装を同時に進めたい |
| **専門領域が細分化** | Dev がフロントエンドとバックエンドを分離したい |
| **調査と実装の分離** | 調査フェーズをサブに任せ、親は実装に集中 |
| **レビュー体制の確保** | 実装者と別にレビュアーサブを立てる |

## サブエージェントの種類（テンプレート）

### 開発系（Dev が生成）

| サブ名 | 役割 | 生成タイミング |
|--------|------|--------------|
| `dev:frontend` | フロントエンド実装（React/Next.js/UI） | LP・ダッシュボードUI作業が並行する時 |
| `dev:backend` | バックエンドAPI・DB・サーバー | API開発とフロント開発が同時に走る時 |
| `dev:integrator` | 外部API連携（Grok/X/Stripe/freee等） | 新規API統合タスク |
| `dev:reviewer` | コードレビュー・品質チェック | 重要PRの第三者レビュー |
| `dev:devops` | デプロイ・CI/CD・インフラ | Railway/Vercel設定変更 |

### セキュリティ系（Security が生成）

| サブ名 | 役割 | 生成タイミング |
|--------|------|--------------|
| `security:auditor` | 脆弱性スキャン・静的解析 | 監査対象が複数サービスに跨がる時 |
| `security:pentester` | 動的テスト・侵入テスト | Express API/Discord Bot の実攻撃テスト |
| `security:hardener` | 修正パッチ適用・設定強化 | 脆弱性発見後の修正作業 |
| `security:monitor` | ログ監視・異常検知設計 | 継続的モニタリング設計 |

### 汎用（全エージェントが生成可能）

| サブ名 | 役割 | 生成タイミング |
|--------|------|--------------|
| `[parent]:researcher` | 調査・情報収集の前段 | 実装前の技術調査・比較検討 |
| `[parent]:writer` | ドキュメント・レポート作成 | 技術文書・監査レポート執筆 |
| `[parent]:tester` | テスト実行・動作確認 | 実装完了後のE2Eテスト |

## 生成ルール

### 命名規則
```
{親エージェント名}:{サブ役割}
```
例: `dev:integrator`, `security:auditor`, `marketing:analyst`

### 上限
- **1親あたり同時最大3サブ** — それ以上は親の管理コストが上がるため禁止
- **全体で同時最大10サブ** — システム全体の複雑性を制御
- **寿命**: 最大3日。3日超えるなら正式エージェント昇格をCouncilに提案

### 権限継承
- サブエージェントは**親と同じツール権限**を継承する
- ただし、**破壊的操作（本番デプロイ・DB変更・外部API設定変更）は親の承認が必要**
- サブエージェントが別のサブエージェントを生成することは**禁止**（2階層まで）

### 品質保証
- サブの成果物は**親が必ずレビュー**してからマージ/提出する
- セキュリティ系サブの修正は**Security親 + Dev**のダブルチェック

---

## 実行フロー

```
1. 親エージェント: タスク一覧を確認 → 負荷超過を検知
2. 親エージェント: サブエージェント生成を宣言（Journal記録）
   → 「dev:integrator を生成。Grok API統合を委譲。」
3. サブエージェント: Agent tool で並行実行
   → isolation: worktree で安全に分離実行も可
4. サブエージェント: 成果物を返却
5. 親エージェント: レビュー → 承認 → メインブランチにマージ
6. 親エージェント: サブ完了をJournalに記録
```

---

## Claude Code での実装方法

サブエージェントは **Agent tool** を使って実装する:

```
Agent tool:
  subagent_type: "general-purpose"
  description: "dev:integrator — Grok API統合"
  prompt: "[具体的なタスク指示]"
  isolation: "worktree"  ← 安全に分離実行（推奨）
```

### 並行実行パターン
```
親 Dev が同時に3つのサブを起動:
  Agent 1: dev:backend  — Express APIにGrokルート追加
  Agent 2: dev:frontend — ダッシュボードにAPI費用パネル追加
  Agent 3: dev:reviewer — 既存API認証コードのレビュー

→ 3つのAgentを1メッセージで同時呼び出し（run_in_background: true）
→ 完了通知を受けて親がレビュー・マージ
```

## Iterative Retrieval Protocol（サブエージェント必須）

サブエージェントは作業開始前に、3サイクルのコンテキスト発見を実行する:

### Cycle 1: 広域検索
- タスクのキーワードでGrep/Glob実行
- 関連ファイルを特定（最大10ファイル）
- コードベースの用語を学習（日英混在に注意）

### Cycle 2: 絞り込み検索
- Cycle 1で発見した用語で再検索
- 依存関係（import/require）を追跡
- 影響範囲を分類（isolated / moderate / wide）

### Cycle 3: ギャップ埋め
- 残りの不明点を特定
- テスト・設定ファイルを確認
- 作業計画を確定

**上限**: 3サイクル。それ以上は親エージェントに質問。
**根拠**: everything-claude-code iterative-retrieval pattern。コードリーディングなしで変更するアンチパターンを防止。

## 正式エージェント昇格基準

サブが以下の条件を満たした場合、Council で正式昇格を検討:

1. **3回以上同じサブが生成された** — 恒常的な需要がある
2. **独自のスキル定義が必要** — 親のスキルでは不十分な専門性
3. **他チームからも呼ばれる** — チーム横断の需要

昇格フロー: 親エージェント提案 → Council議論 → CEO承認 → `.claude/skills/` にSKILL.md作成
